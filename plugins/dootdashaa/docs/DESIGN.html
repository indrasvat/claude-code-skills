▰▰▱ dootdashaa

A replacement for the gsd Claude Code statusline. Bash-first, single-jq-call, p50 ~9ms · p95 ~19ms render budget (measured), repo-aware via git remote (not folder name), with a kartaa-only AWS-credits segment surfaced only when indrasvat/kartaa is the active remote.

2026-05-22 · design report status: awaiting approval ~58 chars at normal load · single line rules: one carrier per signal · zero emoji · NF + plain geometric Unicode only icons: FA + Octicons + Material (NF symbols-only ~50KB)

1 · What you'll see

One line. No duplicate signals. Each piece of information renders as either an icon or text — never both. Icons earn their slot only where they replace a text label (e.g.  AWS replaces the word "aws"). Text stays where it's already as terse as an icon (e.g. 5h is 2 chars; a clock icon would just duplicate the meaning).

3 · Architecture

Two processes, one shared cache dir. The hot path is a pure-read bash script that runs on every assistant message. The cold path is a launchd job at 300s intervals, responsible only for AWS credit refresh (and optional CI-check enrichment later). Stdin payload from Claude Code already provides repo, PR, worktree, rate-limits, cost, and effort — no daemon needed for those.

┌─ HOT PATH (every assistant message · debounced 300ms) ──┐ │ claude code ──► ~/.claude/bin/dootdashaa ──► stdout (single line, 2-line fallback <80 cols) (stdin JSON) │ │ │ │ ONE jq call → @tsv extract 14 fields incl. pr.*, repo.*, │ │ total_input_tokens, used_percentage, cost, 5h/7d, resets_at │ │ git status (5s TTL cache, mtime-invalidated on .git/HEAD+.git/index) │ │ → branch · dirty · ahead/behind │ │ ring-buffer append + 5-sample read → token rate (sparkline opt-in) │ │ $TMPDIR/dootdashaa/rate-<sid> (no JSONL tail) │ │ cat $TMPDIR/dootdashaa/kartaa-aws.line (only when repo matches) │ │ │ └─► printf w/ ANSI escapes → exit 0 always │ honest target: p50 ~15ms · p95 ~25ms · p99 <50ms └────────────────────────────────────────────────────────┘ ┌─ COLD PATH (launchd · StartInterval 300s) ──────────────┐ │ launchd ──────► ~/.claude/bin/dootdashaa-refresh │ │ │ ├─ kartaa-status-aws --line (added in Phase 4) │ │ atomic-write → $TMPDIR/dootdashaa/kartaa-aws.line │ │ │ └─ OPTIONAL (Phase 5+): gh pr checks for CI status │ enrichment beyond what stdin.pr provides (passing/failing) └────────────────────────────────────────────────────────┘

4 · Performance budget

Trigger fires after every assistant message (debounced 300ms; in-flight cancelled). Target: p95 < 25ms. Numbers below are measured against real kartaa session JSONL on macOS 25.3 / bash 3.2 / jq 1.7.

StepBudgetStrategy
Bash startup + read stdin~2 msPreinstalled bash 3.2 on macOS.
One jq call (TSV-extract 14 fields)~4.4 msjq cold-start is real; one call is non-negotiable.
Stat .git/HEAD + .git/index mtime → cache hit?~0.5 msstat -f %m. Cache file git-<sha1(cwd)>.
Git status (cache miss only, ≤5s TTL)~10 ms*git --no-optional-locks status --porcelain=v2 -b. No timeout wrapper (macOS has neither timeout nor gtimeout by default); bounded structurally by 5s TTL — at most one slow git per 5s. *0ms on hit.
Ring-buffer append + 5-sample read (token rate)~1 msAppend "epoch tokens\n" to a 5-line file at $TMPDIR/dootdashaa/rate-<sid>, read with tail -5. Pure bash arithmetic for the rate. Resets on monotonicity break (= /compact).
cat $TMPDIR/dootdashaa/kartaa-aws.line~0.3 msPlain read, no parsing. Skipped entirely outside kartaa.
printf multi-line w/ ANSI escapes~0.5 msPre-composed in bash vars.
p50 (warm git cache)~9 msjq + ring buffer + cat + print.
p95 (one git status hit)~19 msInside the 300ms debounce — comfortable.
p99 (cold everything, sluggish disk)<40 msStill well under 300ms debounce floor.

No timeout dependency. macOS ships no GNU timeout and no gtimeout by default. The 5s git TTL + the 300ms in-flight-cancellation done by Claude Code itself are sufficient bounding — a slow git invocation is naturally pre-empted by the next debounced render.

6 · Repo awareness

Folder name is unreliable — a repo can be cloned under any directory; worktrees have different basenames; rename happens. Claude Code's stdin payload includes workspace.repo.{host,owner,name} as parsed fields ("absent outside a git repository or when no origin remote is configured"). The kartaa check uses those directly:

  jq -r 'if .workspace.repo.owner == "indrasvat" and .workspace.repo.name == "kartaa"
         then "kartaa" else "" end'

  → "kartaa"  ⇒ enable AWS segment
  → ""        ⇒ no AWS segment, no false-positives in non-git dirs

  Worktree of kartaa (different cwd, same origin) ⇒ also matches ✓ (Claude Code resolves the
  remote regardless of which checkout you're in).

Override via DOOTDASHAA_KARTAA_REPO="owner/repo" env var if I ever fork or rename.

Mockup-rendering note: if the NF icons render as tofu in this dashboard, the CDN @font-face fetch was blocked — the codepoint column is the canonical reference. The statusline itself renders against your iTerm/Ghostty's installed NF, which is unaffected.

No-emoji rule (hard)

  • Zero emoji — anywhere. No ⚡ (FA-style emoji), no ‼, no ⏸, no ↻, no 💀. Every state glyph is either a Nerd Font codepoint (PUA, designed monochrome) or a plain Unicode geometric character (block elements, arrows, geometric shapes — also monochrome).
  • Why: emoji render in full colour on macOS via Apple Color Emoji font, which (a) breaks the colour-scheme intent (we choose threshold colour, not the OS), (b) is variable-width across terminals, (c) looks toy-grade next to NF glyphs.
  • Test: any character that renders in colour on iOS Notes is banned.

Compactness conventions

  • Glue values to labels: 5h31% not 5h 31%; $122·5mo not $122 · 5mo. Removes ~30% of horizontal whitespace.
  • Drop redundant prefixes: #222 dies (PR icon replaces it); 45/s not 45 tok/s or 45 t/s (the /s is the unit); $122 not $122 left; 5mo not 161d.
  • Time-of-day economy: durations under 1h → 38m; 1-24h → 2h14 (drop trailing m); 1-60d → 12d; over 60d → 5mo. Boot-recency uses NF rocket prefix: 4h.
  • Hide steady state: medium effort, $0.00 cost, clean+synced git, 7d<20%, AWS OOP=$0, AWS uptime≥24h — all invisible. Surface deviations only.
  • Colour does the alarm work: red text on percentages over 75% (context, 5h limit). No extra characters added to flag urgency.
  • NF icon, plain Unicode, or text — pick one carrier per signal. Never two.

8 · Build plan

Four phases, each shippable on its own. Phase 1 alone replaces gsd usefully.

Phase 1 · MVP (replace gsd)

  • dootdashaa script: model · repo · branch · dirty · ahead/behind · PR + state · context battery icon · cost · 5h/7d rate limits
  • One jq call extracting 14 fields, 5s git mtime cache
  • Width-responsive collapse to 2-line fallback below 80 cols
  • Swap into ~/.claude/settings.json; preserve gsd as a backup

~180 LOC, ~2 hours.

Phase 2 · kartaa AWS segment

  • Add --line flag to kartaa-status-aws emitting $122·5mo
  • launchd plist · 300s interval · explicit PATH
  • dootdashaa-refresh writes atomically to $TMPDIR/dootdashaa/kartaa-aws.line
  • Hot path cats the file only when workspace.repo matches indrasvat/kartaa

~70 LOC + plist, ~1.5 hours.

Phase 3 · token rate (opt-in sparkline)

  • Ring buffer of (epoch, total_input_tokens) tuples written from hot path using stdin's context_window.total_input_tokens
  • 5-sample sliding window, monotonicity-break detects compaction → reset
  • Number always; sparkline gated by DOOTDASHAA_SPARKLINE=1

~50 LOC, ~1 hour.

Phase 4 · CI-check enrichment (optional)

  • Refresher polls gh pr checks for the active branch's PR (if one exists)
  • Writes pass/fail glyph to a cache file the hot path reads
  • Appended after PR state icon: 222·CI

~50 LOC, ~1 hour. Deferred until Phase 1-3 are stable.

Total Phase 1-3: ~300 LOC, ~4.5 hours. Distributed as a Claude Code plugin in the existing indrasvat/claude-code-skills marketplace — see §14 for the install design.

11 · Design invariants

Properties this design guarantees, regardless of session state or future feature additions:

  • Every signal renders through exactly one carrier. Icon, text, or color — never two for the same fact.
  • Zero emoji. Every state glyph is either a Nerd Font codepoint (PUA, designed monochrome) or a plain Unicode geometric character.
  • Hide steady state, surface deviations. Medium effort, $0.00 cost, clean+synced git, 7d limit <20%, AWS OOP=$0, AWS uptime ≥24h — all invisible.
  • Color does the urgency work. No extra characters added to flag a threshold breach; the cell just changes hue (and blinks at ≥95%).
  • Alarms append only when state ≠ healthy. +1-4 chars max; one NF glyph per alarm; never on the steady-state line.
  • Worktree disambiguation via kartaa·wt suffix when workspace.git_worktree is set.
  • Post-/compact graceful degradation — gauge dim-empty, rate/cost hidden, identity + rate-limits + AWS still rendered.
  • Width-responsive fallback — drops to 2 lines below 80 cols, with the gauge as the wrap boundary.

Gotchas captured during verification

Each of these would have shipped to a broken statusline in the first release if skipped during spike work. All now codified for the build step and for anyone reproducing this work later.

GotchaWhy it matters for the buildMitigation in the production script
Literal NF PUA chars get stripped through the editor toolchain Embedding a glyph like U+F407 directly in source produces an empty variable. Both shux and iTerm silently render nothing. Always declare NF glyphs via bash ANSI-C quoting + hex bytes: PR=$'\xef\x90\x87'. Pre-commit hook can xxd-check the file matches expected byte sequences.
iTerm "Use Non-ASCII Font" routing pitfall Default iTerm profile sets "Use Non-ASCII Font" = 1 with "Non Ascii Font" = "Monaco 12". Monaco has no NF glyphs → every PUA codepoint silently routes there and tofus. Operator's machine has it = 0 (lucky), but other installs would break. Statusline install script should defaults read com.googlecode.iterm2 and warn if Use Non-ASCII Font = 1 unless Non Ascii Font is also a Nerd Font. Document in README.
NF v3.x dropped some Octicons codepoints fonttools probe of JetBrainsMonoNerdFontMono-Regular.ttf (v2.304, NF 3.4.0) shows U+F65F (nf-oct-git-pull-request-draft) MISSING. The original design referenced it for the draft-PR state. Draft state surfaces via dim color on the regular PR icon  — aligned with the no-duplicate rule (color carries state, not a separate icon).
printf in bash interprets % as a format specifier First version of render-dootdashaa.sh had printf "...5h31%·2h14...\n" — the %· gets parsed as format spec, prints "invalid format character" errors. Mid-line aborts ruin the rendered output. Mockup strings always go through printf '%s\n' "..." with the format string locked to %s. Avoid printf "literal-with-percent" entirely in the production script.
Bash 3.2 (macOS default) lacks \u escape If the script uses $'', bash 3.2 prints the literal characters uF407. Production needs to run on default macOS bash, not a brew-installed bash 5.x. Use $'\xNN\xNN\xNN' hex-byte form. Works in bash 3.2 and every later version. Each codepoint's UTF-8 encoding tracked in a doc comment next to the variable.
shux rasterizer requires nerd_fonts = true in config Default is true, but if someone (e.g., for ASCII-only environments) sets it false, NF glyphs tofu in shux PNG snapshots. Doesn't affect live terminals — only shux-as-CI tooling. If we use shux for visual regression in CI later, the CI step needs to verify shux config show | grep nerd_fonts = true in setup.
iTerm's Window.async_create() returns before init finishes The returned object's current_tab is None — racing the spike with the iTerm event loop. First spike attempt crashed with AttributeError. Only affects future iterm2-based verification scripts (not the statusline itself). Pattern: asyncio.sleep(0.5) + refresh via async_get_app(connection). Codified in iterm_verify.py.
iTerm screen buffer contains shell-echo lines When reading the buffer to verify "did this codepoint reach the terminal", the line containing the command echo (e.g. printf "MARK[...]") ALSO contains MARK + the literal NF char. Naive grep-for-MARK returns false positives. Only matters for verification scripts. Mitigation: filter out lines starting with shell-prompt or with printf prefix.
macOS screencapture -l can't capture minimized windows If the iTerm window we spawn happens to be minimized (e.g., user has minimize-on-launch behavior), the screencap returns blank. Position the spike window via async_set_frame() immediately after creation. Window correlation via Quartz frame (score < 30) reliably picks the right window in multi-iTerm setups.
Browser preview of this dashboard ≠ ground truth (resolved) The dashboard's <span class="nf"> tags were initially empty (same Write-strip issue as the bash script). What appeared in the browser preview did not match what the statusline renders in iTerm. Resolved by sweep — empty <span class="nf"> tags replaced with HTML numeric character references (&#xF407;, etc.). 57 tags filled, zero empties remaining. Browser preview now matches the terminal screenshots.

14 · Distribution & installation

Ships as a Claude Code plugin in the existing indrasvat/claude-code-skills marketplace. The marketplace already declares one plugin (indrasvat-skills, the 14-skill bundle); dootdashaa is added as a second, independently-installable plugin alongside it.

User-facing install (three lines)

$ claude > /plugin marketplace add indrasvat/claude-code-skills # once per machine > /plugin install dootdashaa@indrasvat-skills # pulls plugin, runs SessionStart hook > /dootdashaa:install # wires ~/.claude/settings.json statusLine

First two are the standard Claude Code plugin install path. The third is a one-time slash command (provided by the plugin itself) that edits ~/.claude/settings.json idempotently to point the user's statusLine at the installed script. Restarting the session picks up the new statusline.

SessionStart hook (plugins/dootdashaa/hooks/hooks.json)

Runs on every session start. Idempotently reconciles two symlinks in ~/.claude/bin/ pointing at ${CLAUDE_PLUGIN_ROOT}/bin/dootdashaa and dootdashaa-refresh. This is the trick that survives plugin updates: ~/.claude/settings.json points at the stable ~/.claude/bin/dootdashaa symlink; the hook re-targets the symlink whenever ${CLAUDE_PLUGIN_ROOT} changes.

{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "\"${CLAUDE_PLUGIN_ROOT}\"/bin/dootdashaa-helper ensure-symlinks"
          }
        ]
      }
    ]
  }
}

dootdashaa-helper ensure-symlinks

#!/usr/bin/env bash
# Reconcile ~/.claude/bin symlinks. Idempotent. Quiet on success.
set +e
here="$(cd "$(dirname "$0")" && pwd)"
mkdir -p "$HOME/.claude/bin"
ln -sfn "$here/dootdashaa"         "$HOME/.claude/bin/dootdashaa"
ln -sfn "$here/dootdashaa-refresh" "$HOME/.claude/bin/dootdashaa-refresh"

# On first install only, hint the user toward /dootdashaa:install
marker="${CLAUDE_PLUGIN_DATA:-$HOME/.claude/plugins/data/dootdashaa}/.first-run-done"
if [ ! -f "$marker" ]; then
  mkdir -p "$(dirname "$marker")"
  touch "$marker"
  echo "dootdashaa installed. Run /dootdashaa:install to wire your statusLine." >&2
fi
exit 0

Why symlinks, not direct paths? ${CLAUDE_PLUGIN_ROOT} changes on every plugin update (Claude Code keeps the prior version dir around for ~7 days, then GCs it). If ~/.claude/settings.json referenced the absolute plugin path, every update would break the statusline. The symlink layer is stable.

/dootdashaa:install slash command

One-time, idempotent. Reads ~/.claude/settings.json; if statusLine is absent, inserts the dootdashaa entry. If a different statusLine is already configured (e.g., gsd), backs it up to statusLine.previous and offers to restore on uninstall.

// commands/install.md (slash-command body, runs jq + writes settings.json)
{
  "statusLine": {
    "type": "command",
    "command": "~/.claude/bin/dootdashaa",
    "padding": 0
  }
}

Update & uninstall

Update/plugin update dootdashaa — pulls latest from indrasvat/claude-code-skills; ${CLAUDE_PLUGIN_ROOT} moves to a new versioned dir; next session's SessionStart hook re-targets ~/.claude/bin/dootdashaa symlink. Zero manual action needed.
Uninstall/dootdashaa:uninstall (removes the statusLine entry from ~/.claude/settings.json; restores any backed-up statusLine.previous) → then /plugin uninstall dootdashaa (Claude Code removes the plugin dir and the symlinks become dangling, but they're harmless once the statusLine entry is gone).
Pi/AWS hostsSame three-line install. Each host's ~/.claude/settings.json is per-host; no shared state to manage.
AWS launchd job (kartaa-only)Separate one-shot command /dootdashaa:install-aws registers ~/Library/LaunchAgents/com.indrasvat.dootdashaa.plist. Optional — only relevant on the operator's main machine where kartaa-status-aws exists.

What the user does NOT have to do

  • No curl … | bash — nothing executes from a URL until the plugin marketplace is explicitly trusted.
  • No chmod +x, no ~/.local/bin PATH wiring — handled by the SessionStart symlink.
  • No hand-editing ~/.claude/settings.json — the /dootdashaa:install command does it idempotently.
  • No git clone, no submodule, no Pi-specific bootstrap script — same flow on every host.
